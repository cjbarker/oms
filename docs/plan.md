# OMS — Old Man Strength (iOS MVP)

## Context

Greenfield repo (`cjbarker/oms`, branch `claude/oms-fitness-app-JfD2K`) currently holds only a LICENSE. Goal: ship a full native iOS MVP of a personalized fitness app that tailors a weekly program to a user's age, weight, health conditions, available equipment, goals, schedule, and experience level. Plans are generated by Claude (BYOK), persisted locally, and each prescribed exercise shows sets/reps/rest plus a YouTube demo link. The app must remain usable when Claude is unavailable (offline / no key / API error) by falling back to a deterministic rule-based engine over a seeded exercise catalog.

## Decisions (locked from clarifying questions)

- **Platform:** iOS only for this cut — Swift + SwiftUI, iOS 17+ (SwiftData, `NavigationStack`, `@Observable`). Android is a later pass that will reuse `exercises.json` and the plan JSON schema.
- **Data:** Local-only. SwiftData for domain entities (profile, plan, sessions). No backend, no account.
- **Videos:** Curated YouTube URL per exercise, opened via `WKWebView` / `SFSafariViewController`.
- **Recommender:** Claude API primary; deterministic rule-based fallback.
- **API key:** BYOK. Stored in iOS Keychain. Entered in Settings; onboarding prompts for it at the end with a "Skip — use built-in generator" option.
- **API endpoint:** Configurable in Settings. Default `https://api.anthropic.com`. User can point it at a compatible proxy or alternate Claude-compatible gateway.
- **Model:** Default `claude-haiku-4-5`, user-configurable in Settings (`claude-haiku-4-5` | `claude-sonnet-4-6` | `claude-opus-4-7`). Same model used for both text plan generation and vision equipment scan (all three support vision).
- **Regeneration:** Auto-regenerate weekly (every Sunday) using last week's logged sessions as context, with a Settings toggle to switch to manual-only (button in the Plan screen).
- **Equipment scan (vision):** New feature — user snaps a photo of their workout space; the app sends the image to Claude (same endpoint/key/model) and gets back a list of detected equipment plus suggested exercises. Detected items merge into `UserProfile.equipment`; next regeneration uses them. Available from onboarding ("Scan my space" as an alternative to manual selection) and from the Settings → Equipment screen.
- **Coach persona:** First step on app launch (before the biometrics survey) the user picks one of three coach personas that colors every AI-generated response — plan descriptions, daily workout intros, exercise cues, encouragement, and the equipment-scan results sheet copy:
  - **Hype** — fun and motivating. Upbeat, warm, celebrates small wins, playful language, emoji-free but enthusiastic.
  - **Drill** — slightly demanding, pushes limits. Direct, no-nonsense, challenges the user, uses imperative verbs, light accountability ("you said 4 days, let's hit 4").
  - **Balanced** — a mix. Warm but firm, explains the "why", escalates intensity when the user is on a streak, softens on rest days.
  Persona is stored on `UserProfile.coachPersona` and fed into every Claude call as part of the system prompt. Changeable any time in Settings → Coach. Fallback engine ships canned copy snippets in each voice for exercise notes and session intros so persona is consistent even offline.
- **HealthKit integration (recovery-aware):** On launch and before each plan/daily generation the app reads (with user permission) **sleep analysis** (last night's total + time in bed), **active energy** and **step count** (yesterday), **workouts** (last 24 h), and — when available — **resting heart rate** and **HRV (SDNN)** from HealthKit. These are packaged into a `RecoverySignals` struct and appended to the Claude user turn so the LLM can dial intensity (e.g. ≤5 h sleep → swap heavy compound for mobility, yesterday's 90-min ride → skip today's legs, resting HR 10 bpm above 7-day baseline → active recovery). A daily "tune today" pass runs on Home open and can downshift the already-generated day without regenerating the whole week. HealthKit is opt-in in Settings and an onboarding step; if denied the app skips the block and the LLM gets an empty `RecoverySignals`. The fallback rule engine also consumes `RecoverySignals` (threshold-based: sleep<6h or yesterdayActiveKcal>500 → reduce today's volume 30 % and bias toward mobility/flexibility).
- **Free-form coaching input:** Final onboarding step before API key is a **"Tell your coach"** free-text field (multi-line, up to 500 chars) where the user can shape the plan in their own words beyond the structured fields — e.g. "3 days/week, 40 min each, focus on core and posterior chain", "I sit at a desk all day so heavy on hip mobility", "no jumping". Stored on `UserProfile.coachingNotes`. Injected verbatim into every Claude call as a separate cached user turn segment. The rule-based fallback parses a limited subset (days/week override, muscle-group keywords, "no X" exclusions) via simple regex so offline generation still respects the hardest constraints.
- **Avoid-list & LLM flagging (negative prompt):** `UserProfile` carries `avoidedExerciseIds: [String]` (hard excludes) and `avoidanceNotes: String` (free-text reasons, e.g. "left shoulder labrum repair 2023, no overhead pressing"). Both are surfaced to Claude in the user turn. The LLM is *also* allowed to raise concerns: for any exercise it would otherwise prescribe but considers risky given the profile, it emits a `flaggedExercises: [{exerciseId, reason, suggestedAlternativeId?}]` entry in the plan JSON rather than silently dropping it. The Plan screen shows a "Coach flagged N items" banner with a per-item sheet: `Accept (include anyway at my risk)` / `Swap in alternative` / `Remove permanently` (adds to the avoid-list). Rule: **no exercise that matches the user's goals is ever silently excluded** — either it appears in the plan, or it appears in the flagged review, or it is on the explicit user avoid-list. The fallback rule engine also emits `flaggedExercises` when contraindication-only exclusions would leave a primary goal unaddressed.

## Key flows

**1. Onboarding → first plan**
1. 8-step survey, with **Step 1 = Coach persona** (Hype / Drill / Balanced, presented as three cards with a one-line sample quote for each). The rest: biometrics → goals → equipment → health → schedule → experience → API key. Writes a `UserProfile` to SwiftData.
2. Final step: prompt for Anthropic API key (Keychain) or "Skip".
3. `PlanGenerator.generate(for: profile)`:
   - If key present: `ClaudeClient.generatePlan(profile, lastWeekSessions: [])` with tool-use forcing the `WeeklyPlan` JSON schema. System prompt cached (includes coaching rules + compact exercise catalog).
   - If no key, offline, or API error: `FallbackRuleEngine.generate(profile)` picks exercises from the seed catalog by goal weighting and equipment filter, excludes contraindicated exercises, splits across `daysPerWeek`, fits `sessionMinutes` via estimated-time-per-set math.
4. Plan saved; user lands on Home.

**2. Daily workout**
- Home shows today's `PlanDay`. Tap → `WorkoutSessionView`: per exercise card with sets/reps/rest, a rest timer, a "Watch demo" button opening `YouTubePlayerView` (WKWebView embed). Each completed set logs reps/weight/RPE. Session persisted on finish.

**3. Weekly regeneration (if enabled)**
- `BGAppRefreshTask` every Sunday: builds a `lastWeekSessions` summary (adherence %, completed volume per muscle group, skipped items) and calls `ClaudeClient.generatePlan(profile, lastWeekSessions:)` for progressive overload and exercise rotation. Manual mode: user taps "Regenerate" on the Plan screen.

**4. Equipment scan (photo → vision LLM)**
- Entry points: Onboarding "Scan my space" button on the Equipment step; Settings → Equipment → "Scan".
- `EquipmentScanView` uses `PHPickerViewController` or `AVFoundation` capture. The image is downscaled to ~1280px and base64-encoded.
- `VisionClient.detectEquipment(imageData:)` calls `{endpoint}/v1/messages` with a `content` array of `[{type: "image", source: {type: "base64", media_type: "image/jpeg", data: ...}}, {type: "text", text: "<catalog-aware prompt>"}]` and a `detect_equipment` tool forcing JSON output.
- The tool schema returns `{detected: [{equipmentId, confidence, notes}], suggestedExerciseIds: [string], unknownItems: [string]}`. The prompt includes the canonical `equipmentId` enum from `exercises.json` so the model returns only valid ids; anything off-enum lands in `unknownItems` and is dropped.
- `ScanResultsSheet` shows detected items with confidence, pre-checked for ≥0.6. User confirms → items merge into `UserProfile.equipment` (union, deduped). A banner offers "Regenerate plan now" if the set changed.
- Failure path: no key / network error / low-confidence empty result → show "We couldn't detect equipment; add it manually" with a link back to the manual picker.

## Claude integration details

- **Endpoint:** `POST {AISettings.endpoint}/v1/messages` — default `https://api.anthropic.com`, user-editable in Settings.
- **Model:** `AISettings.model` (UserDefaults `oms.model`, default `claude-haiku-4-5`); same model drives plan generation and vision scan.
- **Headers:** `x-api-key`, `anthropic-version: 2023-06-01`, `content-type: application/json`
- **Structured output:** Use a `tool` named `submit_weekly_plan` with an `input_schema` matching `docs/plan-schema.json`; `tool_choice: {type: "tool", name: "submit_weekly_plan"}`. Parse `content[0].input` as the plan.
- **Prompt caching:** Mark the system prompt + embedded exercise catalog with `cache_control: {type: "ephemeral"}`. User turn contains only the profile + last-week summary. Cuts cost ~90% on repeat generations.
- **Guardrails in the system prompt:**
  - Leading persona block (from `CoachPersona.voiceGuidelines(_:)`) — e.g. for `drill`: "Voice: direct, demanding, uses imperative verbs. Push the user to commit; acknowledge effort but do not coddle. No emoji. Short sentences." All plan-day intros, exercise notes, and encouragement strings emitted by the model must follow this voice.
  - "You are an ACSM-style coach specializing in training adults 40+..."
  - Hard rules: exclude exercises whose `contraindications` intersect the user's `conditions` (e.g. loaded spinal flexion if `lowBackPain`, high-impact if `kneeOA`, Valsalva-heavy lifts if `hypertension`).
  - Only select `exercise_id`s present in the provided catalog.
  - Day count = `profile.daysPerWeek`; total exercises per workout day must fit `sessionMinutes` at ~6 min/exercise incl. warmup and rest.
  - **Every workout day must contain**: (a) 1–2 `warmup`/mobility items at the start, (b) ≥1 `core` item (planks, anti-rotation, anti-extension, or dead bug family), (c) the main strength/endurance block per the user's primary goal, and (d) 1–2 `cooldown`/flexibility items at the end (static stretches held 30–45 s). Rest days get an optional `activeRecovery` item (walk + 2–3 mobility holds).
  - Balance work (e.g. single-leg stance, SL deadlift regression) included at least 1×/week for users 55+ or with a `balance` goal.
  - Hip mobility: at least 2 hip-tagged mobility items per week (distributed across days, e.g. 90/90, hip CARs, couch stretch, pigeon), raised to 3–4×/week when the user's goal set includes `mobility` or their health step flags `hipStiffness`, `kneeOA`, or `lowBackPain`.
- **Fallback trigger:** network error, non-2xx, invalid JSON, or missing key → `FallbackRuleEngine` with a non-blocking toast.

## Exercise seed library (`Resources/exercises.json`)

~80 entries grouped by category so the engine can always assemble a balanced workout:

- **strength — bodyweight:** pushups (std/incline/knee), squats (air/box), lunges (reverse/walking), glute bridges, single-leg glute bridge, step-ups, pike pushups, inverted rows (table/TRX), wall sit, calf raises.
- **strength — bands:** band pull-aparts, banded rows, banded squats, band face-pulls, banded press.
- **strength — dumbbells:** goblet squat, DB row, DB bench, DB OHP, DB RDL, DB lunge, farmer carry, curls, lateral raise, DB hip thrust.
- **strength — barbell:** back squat, conventional deadlift, bench press, OHP, barbell row, hip thrust (excluded when `lowBackPain` flag is set where applicable).
- **strength — pull-up bar:** pull-ups, chin-ups, scapular pulls, negative pull-ups.
- **core:** front plank, side plank (L/R), dead bug, bird dog, hollow hold, pallof press, suitcase carry, hanging knee raise, bear crawl hold, ab wheel rollout, McGill curl-up.
- **mobility / warmup:** 90/90 hip switches, world's greatest stretch, cat-cow, thoracic rotation (open books), deep squat hold with pry, hip CARs, shoulder CARs, scapular wall slides, leg swings (front-back and lateral), ankle rocks.
- **mobility — hip focus (tagged `hip` on `muscleGroups`):** 90/90 hip switches, hip CARs (controlled articular rotations), frog stretch, fire hydrants, Cossack squats, world's greatest stretch, couch stretch, pigeon pose, figure-4 stretch, seated butterfly, standing lateral lunge to stretch, lying hip circles, banded hip distractions, kneeling hip flexor stretch (half-kneeling), 3-way hip openers. The engine surfaces these whenever the user selects `mobility` as a goal, reports `hipStiffness` / `kneeOA` / `lowBackPain` in the health step, or is age 50+.
- **flexibility / cooldown (static, 30–45 s holds):** seated hamstring stretch, pigeon, child's pose, butterfly, couch stretch, doorway pec stretch, standing quad stretch, figure-4 glute stretch, seated spinal twist.
- **balance:** single-leg stance (eyes open / eyes closed), tandem stance, heel-to-toe walk, single-leg RDL (assisted).
- **endurance:** zone-2 walk, brisk walk intervals, jog, jump rope, stationary cycling, rowing machine, step-ups for time.

Each entry: `id, name, category, muscleGroups, equipment, difficulty (1–5), contraindications, youtubeURL, cues, defaultSets?, defaultReps?, defaultHoldSec?`.

## `README.md` contents

The README will include the following sections, written for a developer who has never seen the repo:

1. **Overview & features** — one-paragraph pitch; feature bullets (personalized weekly plan, daily workouts with sets/reps/video demos, BYOK Claude integration with configurable endpoint/model, photo-based equipment scan, core/hip-mobility/stretching first-class, offline fallback engine, progress tracking).
2. **Screenshots** — placeholder references to `docs/screens/*.png`.
3. **Prerequisites** — macOS 14+, Xcode 15+, iOS 17 SDK; Apple ID (free is enough for on-device dev install; paid Apple Developer Program membership only needed for App Store distribution); optional: an Anthropic API key.
4. **Build** — `git clone git@github.com:cjbarker/oms.git && cd oms && open OMS.xcodeproj`; select the "OMS" scheme; pick an iOS 17 simulator; `⌘B`. CLI equivalent: `xcodebuild -scheme OMS -destination 'platform=iOS Simulator,name=iPhone 15' build`.
5. **Test** — `⌘U` in Xcode, or `xcodebuild test -scheme OMS -destination 'platform=iOS Simulator,name=iPhone 15'`. Lists the test targets and what each covers.
6. **Run on Simulator** — `⌘R` with any iPhone simulator selected. Note that camera features use the simulated photo library since the simulator has no camera.
7. **Run on a local device** — (a) plug the iPhone in via USB and trust the Mac; (b) on the device enable Settings → Privacy & Security → Developer Mode → On (reboot); (c) in Xcode, Signing & Capabilities → Team → select your Apple ID (free provisioning works for personal use; the app re-signs every 7 days unless you're in the paid program); (d) pick the device in the run-destination dropdown; (e) `⌘R`. First launch will require trusting the developer profile under Settings → General → VPN & Device Management.
8. **First-run setup (BYOK)** — complete onboarding, then Settings → AI: paste an Anthropic API key (get one at console.anthropic.com), optionally change the endpoint (default `https://api.anthropic.com`) or model (`claude-haiku-4-5` default, `claude-sonnet-4-6`, `claude-opus-4-7`). Without a key the app still works using the built-in rule-based generator.
9. **Publishing to the App Store** — step-by-step:
   - Enroll in the **Apple Developer Program** ($99/yr) at developer.apple.com.
   - In developer.apple.com → Certificates, Identifiers & Profiles: create an **App ID** with bundle id `com.oms.app` and enable the capabilities the app uses (Push if added later; none required for MVP).
   - In **App Store Connect** (appstoreconnect.apple.com) → My Apps → New App: bundle id `com.oms.app`, primary language, SKU, category **Health & Fitness**.
   - In Xcode → Signing & Capabilities: set **Team** to your paid team, let Xcode manage signing, and confirm a Distribution certificate + App Store provisioning profile are generated.
   - Bump `MARKETING_VERSION` (e.g. `1.0.0`) and `CURRENT_PROJECT_VERSION` (integer, increment every upload) in the project's build settings.
   - `Product → Archive` against "Any iOS Device (arm64)". When the Organizer opens, click **Distribute App → App Store Connect → Upload**, accept defaults, wait for processing in App Store Connect.
   - In App Store Connect → App Privacy: declare data types collected (none leave the device by default; prompt payloads are sent to the user-configured AI endpoint), declare Camera and Photo Library usage tied to the equipment-scan feature.
   - Fill out **version information**: screenshots for the required device sizes (6.7" and 6.1" iPhone at minimum), promo text, description, keywords, support URL, privacy policy URL, age rating (likely 12+ due to fitness content).
   - **Reviewer notes**: explain the BYOK model and provide a temporary Anthropic key reviewers can use (or instructions to use the built-in fallback path so review doesn't require a key).
   - Submit for review; typical turnaround 1–2 days. Common rejection causes to pre-empt: missing `NSCameraUsageDescription` / `NSPhotoLibraryUsageDescription` strings, privacy-manifest omissions, placeholder screenshots.
10. **Privacy** — `NSCameraUsageDescription` ("OMS uses the camera to identify equipment in your workout space"), `NSPhotoLibraryUsageDescription` ("OMS can analyze a photo of your space to detect equipment"), Keychain storage for the API key, plain HTTPS to the user-configured endpoint, no analytics SDKs.
11. **Troubleshooting** — common Xcode signing errors, simulator camera caveats, "invalid API key" vs "endpoint unreachable" error messages surfaced in the app.
12. **License** — points at top-level `LICENSE` (MIT).

## Verification

1. **Build:** Open `OMS.xcodeproj` in Xcode 15+, target "iPhone 15 (iOS 17)" simulator, `⌘R`. App launches to onboarding.
2. **Onboarding with key:** Complete all 7 steps, enter a valid Anthropic key. Expect a "Generating your plan…" spinner, then the Home screen with today's workout populated. Network call visible in Console logs.
3. **Onboarding without key (fallback):** Repeat onboarding, Skip the key step. Plan still generates within ~50 ms via `FallbackRuleEngine`.
4. **Contraindication check:** Re-onboard selecting `lowBackPain`. Inspect the resulting plan — no conventional deadlifts, sit-ups, or loaded good-mornings should appear (both in Claude and fallback paths).
4a. **Balanced workout check:** Inspect any workout day — it must contain at least one warmup/mobility item, at least one core item, the main strength/endurance block, and at least one cooldown/flexibility item. Verify this holds for both the Claude path and the fallback engine. Users 55+ or with `balance` goal must see at least one balance exercise per week.
5. **Equipment filter:** Re-onboard with `equipment = [none]`. All prescribed exercises must have `equipment = [none]`.
6. **Workout session:** Tap today's workout → each exercise shows sets/reps/rest, "Watch demo" opens YouTube in-app. Complete a session → appears in Progress tab with correct volume.
7. **Weekly regen (manual):** Plan screen → "Regenerate". Confirms it calls Claude with last week's session data in the user turn (log the request body in debug builds).
8. **Settings:** Change model to `claude-sonnet-4-6`, regenerate → network request `model` field matches. Toggle auto-regen off → `WeeklyRegenScheduler` unregisters the `BGAppRefreshTask`.
9. **Equipment scan:** Settings → Equipment → Scan. Pick a sample photo (a kettlebell + bench). Expect the results sheet to list both items with confidence ≥0.6, pre-checked. Confirm → `UserProfile.equipment` contains `kettlebell` and `bench`. Tap "Regenerate" → next plan uses them. With no key, verify the scan button surfaces a "Set up AI in Settings" prompt instead of failing silently.
10. **Custom endpoint:** In Settings change AI endpoint to an invalid URL → both plan regen and scan fail gracefully with a clear error and fallback (plan) / manual-add suggestion (scan).
10a. **Coach persona:** Complete onboarding three times picking Hype, Drill, and Balanced. Compare the generated plan's day intros and exercise notes — Hype is upbeat and celebratory, Drill is terse and demanding, Balanced blends warmth with firm cues. Switch persona in Settings → Coach → Drill, regenerate, confirm voice updates. Offline (no key) path: confirm fallback engine's canned strings also reflect the selected persona.
11. **Unit tests:** `⌘U` runs the five test files above; all pass. Key tests: fallback always returns `daysPerWeek` workout days, never includes a contraindicated exercise, every catalog entry decodes and has a non-empty `youtubeURL`, vision client ignores `equipmentId` values not in the catalog enum.
